# i18n — Reglas

## Navegación

**Para enlaces internos, usar siempre `Link` de `@/i18n/navigation`**:
```ts
import { Link } from '@/i18n/navigation';          // ✓
import Link from 'next/link';                       // ✗ no localizado
```

**Para router.push/replace, usar siempre el wrapper**:
```ts
import { useRouter } from '@/i18n/navigation';     // ✓
import { useRouter } from 'next/navigation';        // ✗ no localizado
```

**Para leer el pathname actual (sin prefijo de locale)**:
```ts
import { usePathname } from '@/i18n/navigation';   // ✓ devuelve /users
import { usePathname } from 'next/navigation';      // ✗ devuelve /en/users
```

## Rutas

**Las rutas en `config/routes.ts` NO incluyen prefijo de locale**:
```ts
routes.dashboard = '/'       // ✓
routes.users.list = '/users' // ✓
routes.dashboard = '/en/'    // ✗
```

Los wrappers de `@/i18n/navigation` añaden el prefijo automáticamente.

## Strings — Regla general

**Cualquier string visible en UI debe ir en un archivo de traducción. Nunca hardcodear strings en componentes.**

- Strings de un feature → `src/features/{feature}/i18n/{locale}.json`
- Strings transversales → `src/messages/{locale}/common.json`
- Los namespaces coinciden con los keys en `loadMessages` de `src/i18n/request.ts`

## Client vs Server Components

**Client Components** — usar `useTranslations`:
```ts
'use client';
import { useTranslations } from 'next-intl';
const t = useTranslations('users');  // namespace del feature
return <h1>{t('title')}</h1>;
```

**Server Components** — usar `getTranslations` (es async):
```ts
import { getTranslations } from 'next-intl/server';
const t = await getTranslations({ locale, namespace: 'reports' });
```

**Regla clave**: No añadir `'use client'` a un componente solo para poder usar `useTranslations`. Si el componente no necesita ser cliente por otras razones, convertirlo en async server component y usar `getTranslations`.

## Naming conventions — jerarquía de claves

```
common.actions.*          — save, cancel, delete, edit, create, retry, goHome
common.status.*           — loading, error, empty
common.navigation.*       — searchPlaceholder, notifications
common.sidebar.sections.* — workspace, ui
common.sidebar.items.*    — dashboard, analytics, reports, users, settings, uiOverview...
common.sidebar.actions.*  — expand, collapse, logOut
common.userMenu.*         — profile, settings, signOut, signingOut
common.theme.*            — switchToLight, switchToDark
common.metadata.*         — title, description

auth.login.*              — title, subtitle, welcomeBack, form.*, errors.*, validation.*, demo.*
users.*                   — title, subtitle, actions.*, ...
```

Las claves siguen `namespace.categoria.elemento` (3 niveles máximo salvo casos especiales).

## Patrón sidebar.config.ts — labels como claves

Los `label` en `sidebar.config.ts` son claves de traducción (relativas al namespace `common`):

```ts
// ✓ correcto
{ type: 'link', label: 'sidebar.items.dashboard', href: routes.dashboard }

// ✗ nunca — string literal
{ type: 'link', label: 'Dashboard', href: routes.dashboard }
```

El componente que renderiza resuelve la clave con `useTranslations('common')`. El config sigue siendo
una constante estática.

## Zod schemas con mensajes traducidos

Crear una factory que recibe los mensajes como parámetros:

```ts
// lib/validators/entity.schema.ts
export function createEntitySchema(messages: { nameRequired: string }) {
  return z.object({ name: z.string().min(1, messages.nameRequired) });
}

// En el componente (client)
const t = useTranslations('users');
const schema = createEntitySchema({ nameRequired: t('validation.nameRequired') });
const form = useForm({ resolver: zodResolver(schema) });
```

## Interpolación de parámetros

```json
{ "welcomeBack": "Welcome back, {name}!" }
```
```ts
t('welcomeBack', { name: user.name })
```

## Metadata localizada

```ts
// [locale]/layout.tsx
export async function generateMetadata({ params }) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'common' });
  return {
    title: { default: t('metadata.title'), template: `%s · ${t('metadata.title')}` },
    description: t('metadata.description'),
  };
}
```

## Añadir un namespace nuevo

1. Crear `src/features/{feature}/i18n/en.json` y `es.json`
2. Añadir la importación en `loadMessages` en `src/i18n/request.ts`
3. Añadir al objeto retornado por `loadMessages`

## Estructura de archivos de traducción

```json
{
  "title": "...",
  "subtitle": "...",
  "actions": { "create": "..." },
  "validation": { "nameRequired": "..." }
}
```

Evitar namespaces planos sin anidación — los strings de acciones van bajo `"actions"`.

## Cookie

El locale se persiste en la cookie `NEXT_LOCALE`. next-intl la gestiona automáticamente.
No leer ni escribir esta cookie manualmente — usar `LanguageSwitcher` de `@/components/i18n/language-switcher`.

## Texto enriquecido (`t.rich`)

Para traducciones que mezclan prosa con `<code>`, `<strong>` u otros tags inline, usar `t.rich`
en lugar de concatenar JSX con strings:

```tsx
const code = (chunks: React.ReactNode) => (
  <code className="font-mono bg-[var(--surface-raised)] px-1 py-0.5 rounded">{chunks}</code>
);
{t.rich('sections.persistence.footnote', { code })}
```

JSON:
```json
"footnote": "Store key: <code>nexdash-sidebar</code> in localStorage."
```

## ICU escape — llaves literales

Si el texto debe mostrar llaves que NO son placeholders (p. ej. `{sectionId}`), envolver en
single quotes para activar modo literal de ICU:

```json
"footnote": "ID computed as <code>'{sectionId}.{kebab-label}'</code>"
```

Resultado renderizado: `ID computed as <code>{sectionId}.{kebab-label}</code>`.

## Patrón split server/client para `generateMetadata`

Las páginas que necesitan ser cliente (interactividad, state) **y** quieren metadata localizada
deben dividirse en dos archivos:

```
page.tsx              ← server, async, exporta generateMetadata. Renderiza <Content/>
{name}-content.tsx    ← 'use client', contiene la UI completa
```

```tsx
// page.tsx
import { getTranslations } from 'next-intl/server';
import { ToastsContent } from './toasts-content';

export async function generateMetadata({ params }) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'uiShowcase' });
  return { title: t('toasts.metadata.title') };
}

export default function ToastsPage() {
  return <ToastsContent />;
}
```

Patrón ya en uso en `settings/page.tsx`, `reports/page.tsx`, y todas las páginas `/ui/*` que
necesitan interactividad.

## Convención específica de la sección /ui (showcase)

Las páginas bajo `/ui/*` documentan el sistema de diseño. Política de traducción híbrida:

**No se traducen**:
- Nombres propios del design system (Toast, Sidebar, Skeleton, Empty state)
- Jerga universal (Variants, Props, API, Methods, States)
- Identificadores en `PropsTable` (nombres de props, tipos, defaults)
- Snippets de código visibles dentro de `code={...}`

**Sí se traducen**:
- Headers descriptivos, subtítulos, prosa
- Labels y mensajes de demos (los strings que el usuario ve cuando interactúa)
- Descripciones de cada `ShowcaseSection` y de cada prop (columna Description)
- Headers de columnas en tablas

### Duplicación intencional snippet/demo

```tsx
<ShowcaseDemo
  title={t('demos.success.title')}
  // Snippet visible: SIEMPRE en inglés (referencia copy-paste)
  code={`toast.success('User created successfully');`}
>
  {/* Demo funcional: traducida al locale activo */}
  <Button onClick={() => toast.success(t('demos.success.message'))}>
    {t('demos.success.label')}
  </Button>
</ShowcaseDemo>
```

### `PropsTable` con descripciones traducidas

El array `props: PropDoc[]` se construye **dentro del componente** (no a nivel de módulo) para
poder usar `t()` en cada `description`:

```tsx
const t = useTranslations('uiShowcase.toasts');
const props: PropDoc[] = [
  { prop: 'title', type: 'string', description: t('props.title'), required: true },
];
```

### Sección "Localization" al final

Cada página de `/ui` termina con un `<ShowcaseSection title={tNote('title')}>` que enlaza a
`/ui/i18n`. Reusar la clave `uiShowcase.localizationNote.{title,description,linkLabel}`.

## Limitaciones conocidas

- **Skeleton aria-labels** (`Loading table`, etc.) se mantienen en inglés. Razón: añadir
  `useTranslations` a estos componentes presentacionales los forzaría a `'use client'` solo
  para un string brevemente audible. Documentado como deuda técnica aceptable.
