# Internationalization (i18n)

Sistema de internacionalización basado en [next-intl](https://next-intl.dev/) v4 con App Router.

---

## Arquitectura

```
src/
  config/
    i18n.ts               ← Configuración base: locales, defaultLocale, LOCALE_COOKIE
  i18n/
    routing.ts            ← defineRouting (locales, prefix, cookie, detection)
    navigation.ts         ← Link, useRouter, usePathname, redirect localizados
    request.ts            ← getRequestConfig: carga mensajes por locale + feature
  messages/
    en/common.json        ← Strings transversales en inglés
    es/common.json        ← Strings transversales en español
  app/
    layout.tsx            ← Root layout: lee locale de cookie para <html lang>
    [locale]/
      layout.tsx          ← NextIntlClientProvider + Providers
      (auth)/...          ← Rutas públicas: /en/login, /es/login
      (dashboard)/...     ← Rutas protegidas: /en/, /es/users, etc.
  features/
    {feature}/
      i18n/
        en.json           ← Namespace del feature en inglés
        es.json           ← Namespace del feature en español
  proxy.ts                ← Middleware: intl primero, auth después
```

## Locales

Definidos en `src/config/i18n.ts`:
- `en` (inglés) — locale por defecto
- `es` (español)

Para añadir un nuevo locale:
1. Añadir a `locales` en `src/config/i18n.ts`
2. Crear `src/messages/{locale}/common.json`
3. Crear `src/features/{feature}/i18n/{locale}.json` para cada feature
4. `localeLabels` y `localeFlags` en `src/config/i18n.ts`

## Routing

`localePrefix: 'always'` → todas las URLs llevan prefijo:
- `/` → redirige a `/en/` (o `/es/` por cookie/Accept-Language)
- `/en/users`, `/es/users` → ambas válidas
- `/users` → redirige a `/en/users`

## Persistencia de locale

Cookie `NEXT_LOCALE` (configurable en `routing.ts`). next-intl la gestiona automáticamente:
- Al visitar `/en/*` → establece `NEXT_LOCALE=en`
- Al visitar `/es/*` → establece `NEXT_LOCALE=es`
- En visitas posteriores sin prefijo → usa la cookie

## Namespaces por feature

Los mensajes se organizan por feature, cada uno en su propio JSON:

| Feature | Archivo | Namespace en useTranslations |
|---|---|---|
| Global | `src/messages/{locale}/common.json` | `'common'` |
| Auth | `src/features/auth/i18n/{locale}.json` | `'auth'` |
| Dashboard | `src/features/dashboard/i18n/{locale}.json` | `'dashboard'` |
| Analytics | `src/features/analytics/i18n/{locale}.json` | `'analytics'` |
| Users | `src/features/users/i18n/{locale}.json` | `'users'` |
| Reports | `src/features/reports/i18n/{locale}.json` | `'reports'` |
| Settings | `src/features/settings/i18n/{locale}.json` | `'settings'` |
| UI Showcase | `src/features/ui-showcase/i18n/{locale}.json` | `'uiShowcase'` |

Los strings de un feature van en el JSON de ese feature, no en `common.json`.
`common.json` es solo para strings verdaderamente transversales (acciones genéricas, meses, días).

## Cómo usar traducciones

### En Client Components

```tsx
'use client';
import { useTranslations } from 'next-intl';

export function MyComponent() {
  const t = useTranslations('dashboard');
  return <h1>{t('title')}</h1>;
}
```

### En Server Components

```tsx
import { getTranslations } from 'next-intl/server';

export default async function Page() {
  const t = await getTranslations('users');
  return <h1>{t('title')}</h1>;
}
```

### Con parámetros interpolados

```json
{ "greeting": "Welcome back, {name}!" }
```
```tsx
t('greeting', { name: user.name })
```

## Navegación localizada

**Siempre usar los wrappers de `@/i18n/navigation`**, nunca `next/link` ni `next/navigation` directamente:

```tsx
import { Link, useRouter, usePathname } from '@/i18n/navigation';

// Link — añade prefijo de locale automáticamente
<Link href="/users">Users</Link>  // → <a href="/en/users">

// Router — router.push sin prefijo; el wrapper lo añade
const router = useRouter();
router.push('/settings');  // navega a /en/settings

// Pathname — devuelve path SIN prefijo de locale
const pathname = usePathname();
// URL actual: /en/users → pathname: /users
```

**Las rutas en `config/routes.ts` NO llevan prefijo de locale**:
```ts
routes.dashboard = '/'       // ✓
routes.users.list = '/users' // ✓
routes.dashboard = '/en/'    // ✗ incorrecto
```

## Composición del proxy (middleware)

`src/proxy.ts` compone el middleware de next-intl con la lógica de auth:

1. **Rutas excluidas** (`/api`, `/_next`, archivos estáticos) → `NextResponse.next()` directo
2. **intlMiddleware** → resuelve locale, redirige si falta el prefijo (e.g. `/` → `/en/`)
3. **Extracción de locale** → desde la URL efectiva (destino de la redirección si aplica)
4. **Auth check**:
   - Sin sesión + ruta protegida → redirect a `/{locale}/login`
   - Con sesión + ruta login → redirect a `/{locale}/`
5. **Return intlResponse** → si auth pasa, devolver la respuesta del intl middleware

El orden intl→auth evita la doble redirección: se detecta el locale antes de redirigir al login,
por lo que el redirect ya lleva el prefijo correcto (`/en/login`, no `/login` → `/en/login`).

## Verificación end-to-end

Flujos verificados con `npm run build` (34 páginas generadas: 17 rutas × 2 locales):

1. `/` → redirect por middleware a `/en/` (o `/es/` por cookie)
2. `/en/` → carga dashboard con `useTranslations('dashboard').title = "Dashboard"`
3. `/es/` → carga dashboard con `useTranslations('dashboard').title = "Panel"`
4. Sin sesión → redirect a `/{locale}/login`
5. Con sesión en `/en/login` → redirect a `/en/`

## Convención de la sección /ui (showcase)

Las páginas bajo `/ui/*` documentan el sistema de diseño. La política de traducción es híbrida:

**Permanecen en inglés** (no se traducen):
- Nombres de componentes del design system: Toast, Empty state, Error state, Sidebar, Dropdown, Popover, Dialog
- Jerga universal: Variants, Props, API, Methods, States
- Términos técnicos por convención: Hook, Component, Utility, Token
- Identificadores en `PropsTable`: nombres de props, tipos, valores por defecto
- Snippets de código visibles (referencia copy-paste)

**Se traducen**:
- Títulos de página y subtítulos descriptivos
- Descripciones de cada `ShowcaseSection`
- Labels y mensajes de los botones de las demos
- Descripciones de cada prop (columna Description)
- Headers de columnas en tablas (`PropsTable` y otras)
- Textos de status y notas (Coming soon, soon, Localization)

### Snippets en inglés vs demos en idioma activo

Las demos del showcase introducen una **duplicación intencional**: el snippet de código visible
permanece en inglés, mientras que la demo (botón funcional) usa `t()` y muestra el texto en el
locale activo. Razón: el snippet es educativo (cómo se escribe el código), la demo es funcional
(cómo se ve al usuario).

```tsx
<ShowcaseDemo
  title={t('demos.success.title')}
  // Snippet visible: siempre en inglés
  code={`toast.success('User created successfully');`}
>
  {/* Demo funcional: traducida al locale activo */}
  <Button onClick={() => toast.success(t('demos.success.message'))}>
    {t('demos.success.label')}
  </Button>
</ShowcaseDemo>
```

### `PropsTable` con descripciones traducidas

El array `props: PropDoc[]` se construye **dentro del componente cliente**, no a nivel de módulo,
para poder usar `t()` en cada `description`. Los demás campos (prop, type, default) son código y
permanecen en inglés.

```tsx
const t = useTranslations('uiShowcase.toasts');
const props: PropDoc[] = [
  { prop: 'title', type: 'string', description: t('props.title'), required: true },
  { prop: 'duration', type: 'number', default: 'variant-dependent', description: t('props.duration') },
];
```

### Plural ICU para contadores

`next-intl` soporta nativamente el formato ICU. Para contadores con plural:

```json
{ "componentCount": "{count, plural, one {# component} other {# components}}" }
```
```tsx
t('overview.componentCount', { count })
```

### Sección "Localization" al final de cada página

Cada página de showcase termina con un `<ShowcaseSection>` que recuerda al desarrollador que en
producción los strings deben envolverse en `t()`, con un link a `/ui/i18n`. El texto vive en
`uiShowcase.localizationNote.{title,description,linkLabel}` y se reutiliza en todas las páginas.

### Metadata de la pestaña

Cada página tiene `generateMetadata` localizada. Los nombres propios del design system mantienen
su forma original ("Toasts" en ambos locales); los conceptos genéricos se traducen ("Empty states"
→ "Estados vacíos", "Error states" → "Estados de error").

Para soportar `generateMetadata` en páginas que necesitan ser cliente (interactividad), se divide
en dos archivos: `page.tsx` server component con `generateMetadata` que renderiza un
`{name}-content.tsx` `'use client'`. Mismo patrón ya en uso en `settings/page.tsx` y
`reports/page.tsx`.

## Cómo añadir un idioma nuevo

Esta guía describe el proceso paso a paso para añadir un tercer idioma (ejemplo: francés `fr`).

### 1. Registrar el código en `src/config/i18n.ts`

```ts
export const locales = ['en', 'es', 'fr'] as const;

export const localeLabels: Record<Locale, string> = {
  en: 'English',
  es: 'Español',
  fr: 'Français',
};

export const localeFlags: Record<Locale, string> = {
  en: '🇬🇧',
  es: '🇪🇸',
  fr: '🇫🇷',
};
```

`Locale` se infiere del array, así que TypeScript te avisará en cualquier sitio que dependa
de un literal `'en' | 'es'` cuando sea necesario actualizarlo.

### 2. Crear los archivos de traducción

Por cada feature + el namespace `common`, crear el archivo en el nuevo locale:

```
src/messages/fr/common.json                 ← copiar de en/common.json y traducir
src/features/auth/i18n/fr.json              ← idem
src/features/dashboard/i18n/fr.json
src/features/analytics/i18n/fr.json
src/features/users/i18n/fr.json
src/features/reports/i18n/fr.json
src/features/settings/i18n/fr.json
src/features/ui-showcase/i18n/fr.json
```

La forma más rápida: copiar los archivos `en` como punto de partida y traducir el contenido.
Mantener intactas las claves y los placeholders (`{name}`, `{count}`, `<code>...</code>`,
formatos ICU como `{count, plural, one {...} other {...}}`).

### 3. (No es necesario tocar `request.ts`)

`src/i18n/request.ts` carga dinámicamente por locale: `import('../messages/${locale}/common.json')`
y `import('../features/.../i18n/${locale}.json')`. Mientras los archivos existan en disco, se
cargan automáticamente.

### 4. Verificar el LanguageSwitcher

`LanguageSwitcher` lee `locales` desde `@config/i18n` y muestra todos los idiomas registrados.
No requiere cambios.

### 5. Notas según el idioma

- **RTL** (árabe, hebreo): añadir `dir="rtl"` en `<html>` cuando el locale sea RTL. Esto requiere
  modificar `src/app/layout.tsx` para leer el locale y mapearlo a `dir`. No implementado aún —
  documentar como deuda si se añade un idioma RTL.
- **Calendarios no gregorianos**: `useFormatter` y `Intl.DateTimeFormat` se ajustan
  automáticamente al locale; revisar el formato resultante.
- **Plurales con más formas**: árabe tiene 6 formas, polaco 4. ICU MessageFormat las soporta
  (`zero`, `one`, `two`, `few`, `many`, `other`); ampliar las claves plural en consecuencia.

### 6. Probar el flujo completo

1. `npm run build` — verifica que las nuevas claves no rompen el build.
2. `npm run dev`, navegar a cualquier página y abrir el LanguageSwitcher: el nuevo idioma debe
   aparecer.
3. Click en el idioma nuevo: la URL cambia a `/fr/...`, la cookie `NEXT_LOCALE=fr` se establece,
   y todos los textos se renderizan en francés.
4. Recargar: la cookie persiste, la URL sigue en `/fr/...`.
5. Visitar una ruta sin prefijo (`/`): el middleware redirige a `/fr/` por la cookie.
6. Borrar la cookie y volver a entrar: el middleware detecta el `Accept-Language` del navegador.

## Glosario — términos que NO se traducen

Para mantener coherencia entre locales, los siguientes términos permanecen en inglés en toda la app:

| Categoría | Términos | Razón |
|---|---|---|
| Componentes del design system | `Toast`, `Skeleton`, `Sidebar`, `Empty state`, `Error state`, `Dropdown`, `Popover`, `Dialog`, `Card`, `Badge`, `Avatar`, `Tooltip` | Nombres propios reconocibles transversalmente |
| Jerga universal | `Variants`, `Props`, `API`, `Methods`, `States`, `Hook`, `Component`, `Utility`, `Token` | Convenciones del ecosistema React/web |
| Identificadores de código | nombres de props, tipos, métodos, valores por defecto en `PropsTable` | Son código, no prosa |
| Snippets de código en /ui | strings de ejemplo dentro de los `<code>` blocks | Referencia copy-paste educativa |
| Headers de pestaña con nombre propio | `Toasts`, `Sidebar`, `i18n` | Si es nombre propio del componente, queda |

Los términos descriptivos sí se traducen ("Empty states" → "Estados vacíos") cuando son
conceptos genéricos en lugar de nombres propios.

### Texto enriquecido (`t.rich`) con tags inline

Para descripciones que mezclan prosa con `<code>` o `<strong>`, usar `t.rich` con callbacks:

```tsx
const code = (chunks: React.ReactNode) => (
  <code className="font-mono bg-[var(--surface-raised)] px-1 py-0.5 rounded">{chunks}</code>
);
{t.rich('sections.persistence.footnote', { code })}
```

JSON:
```json
{ "footnote": "Store key: <code>nexdash-sidebar</code> in localStorage." }
```

### ICU escape para llaves literales

Para texto con llaves que no son placeholders (p.ej. mostrar `{sectionId}` literalmente),
envolverlo en single quotes — modo literal de ICU:

```json
"footnote": "ID computed as <code>'{sectionId}.{kebab-label}'</code>"
```

Resultado renderizado: `ID computed as <code>{sectionId}.{kebab-label}</code>`.

## Limitaciones conocidas

- **Skeleton aria-labels** (`Loading table`, `Loading list`, etc.) permanecen en inglés. Razón
  arquitectónica: añadir `useTranslations` forzaría a estos componentes presentacionales a ser
  client components, solo para un string audible brevemente a usuarios de screen reader.
  Documentado como deuda técnica aceptable; abrir un ticket si llegan reportes accesibles.
- **No hay soporte RTL todavía**. Si se añade un idioma RTL (árabe, hebreo), el `<html dir>` debe
  fijarse desde `src/app/layout.tsx` leyendo el locale.
